# 节点交互模型重设计方案

> 将 How / Why 层从自由输入改为选项题，System 层改为卡片拖拽构建连接图，原问回响保留自由输入但改为 AI 软判定。降低用户输入摩擦，把唯一的自由表达集中在最后的高潮环节。

文档创建日期：2026-06-19
状态：设计待实现

---

## 1. 设计动机

### 1.1 当前问题

| 问题 | 表现 | 后果 |
|------|------|------|
| 输入量过大 | How / Why / System 三层 + 原问回响共 4 次自由输入 | 用户疲劳，中途放弃 |
| 强校验挫败感 | AI 判定"还缺一点"→ 用户反复修改 | 产生"我答不对"的焦虑 |
| 交互形式雷同 | 四层都是对话框 + 输入框 | 缺乏节奏变化，体验单调 |
| 认知活动分散 | 每层都要生成表达 | 用户精力被稀释，深度思考反而变浅 |

### 1.2 核心思路

> 前四层降低摩擦、保留认知活动，把唯一的自由表达集中在最后的高潮。

- How / Why：用选项题替代自由输入，保留"判断 + 选择"的认知活动，消除打字负担
- System：用卡片拖拽构建连接图，交互形式差异化，更有游戏感
- 原问回响：保留自由输入，但改为 AI 软判定，覆盖核心要点即通过

### 1.3 新旧对比

| 维度 | 旧设计 | 新设计 |
|------|--------|--------|
| 用户输入量 | 4 层都要打字 | **Why 层 1 次 + 原问回响 1 次，共 2 次** |
| 挫败感来源 | AI 强判定"还缺一点"反复修改 | 选错给反馈但不卡住；输入环节软判定不卡人 |
| 交互多样性 | 4 层都是对话框 + 输入框 | 阅读 → 选择 → 选择+输入 → 拖拽 → 写作 |
| 认知活动 | 每层都要生成表达 | 前三层判断 + 组织，最后两层逐步过渡到综合表达 |
| 通过难度 | 每层都可能卡住 | 前三层必过，输入环节软判定 |

## 2. 整体流程

```
What 层（不变）
  对话 + 三卷轴阅读 + 选择关键卷轴
        ↓
How 层
  3 道 4选1 单选题，选错重新选择直到正确
        ↓
Why 层
  3 道 4选1 单选题 + 1 道开放输入（AI 软判定）
        ↓
System 层
  卡片拖拽构建连接图，建立 ≥3 条有效连接
        ↓
原问回响
  自由输入 + AI 软判定，覆盖 ≥2 条标准即通过
```

## 3. How 层：4选1 单选题

### 3.1 界面

```
┌────────────────────────────────────────────────┐
│  ⚙️ How · 理解层 · 认知革命              1/3   │
├────────────────────────────────────────────────┤
│                                                │
│  🧙 老学者："虚构故事是如何让陌生人             │
│            开始协作的？"                        │
│                                                │
│  ○ A. 因为故事让人类变得更聪明                  │
│                                                │
│  ○ B. 因为共同相信一个故事，陌生人之间          │
│       有了信任基础，可以合作                    │
│                                                │
│  ○ C. 因为故事能记录历史，避免重复犯错          │
│                                                │
│  ○ D. 因为故事能娱乐，让人心情好                │
│                                                │
│                              [ 确认选择 → ]     │
└────────────────────────────────────────────────┘
```

### 3.2 题量与完成条件

- 每层 **3 道题**
- 选错 → 重新选择直到正确
- 3 题全对 → 当前深度 `completed`

### 3.3 选项设计原则

**关键：4 个选项都有部分道理，没有明显荒谬的选项。**

| 选项角色 | 设计要求 | 来源 |
|---------|---------|------|
| 正确选项 | 最完整、最准确的解释 | Scope 文档对应深度的掌握标准 |
| 干扰项 1 | 常见误解的表述 | Scope 文档「常见误解」 |
| 干扰项 2 | 不完整理解（只说对了一半） | 部分正确但缺关键要素 |
| 干扰项 3 | 相关但未触及核心 | 表面相关但答非所问 |

### 3.4 选错反馈递进

| 第几次错 | 反馈内容 | 示例 |
|---------|---------|------|
| 第 1 次 | 简短提示，不透露答案 | "再想想，这个选项不够完整。" |
| 第 2 次 | 具体提示，指向关键要素 | "关键在于'信任基础'——陌生人之间需要什么才能合作？" |
| 第 3 次 | 直接展示正确答案 + 解释 | "正确答案是 B。因为共同相信一个故事，陌生人之间才有了信任基础……" |

第 3 次后直接进入下一题，不继续卡住用户。

### 3.5 题目数据来源

每道题按以下结构存储：

```typescript
interface ChoiceQuestion {
  question: string;           // 题干
  options: ChoiceOption[];    // 4 个选项
  correctIndex: number;       // 正确选项索引
  feedback: {
    hint1: string;            // 第1次错的提示
    hint2: string;            // 第2次错的提示
    explanation: string;      // 第3次错的完整解释
  };
  scopeRef: string;           // 对应 Scope 文档的掌握标准编号
}

interface ChoiceOption {
  text: string;
  misconceptionRef?: string;  // 如果是干扰项，对应 Scope 文档的常见误解
}
```

### 3.6 How 层出题示例（认知革命）

**对应 Scope 标准②：说明"虚构如何促进协作"**

| 题号 | 题干 | 正确选项核心 | 干扰项来源 |
|------|------|------------|-----------|
| 1 | 虚构故事是如何让陌生人开始协作的？ | 共同相信→信任基础→合作 | 误解"更聪明""记录历史""娱乐" |
| 2 | 下面哪个例子最能体现"虚构促进协作"？ | 一群陌生人因共同信仰放下武器 | 误解"工具""火""资源" |
| 3 | 邓巴数字（150人）被突破的关键是什么？ | 虚构故事让超出熟人范围的人也能互信 | 误解"脑容量""工具""语言复杂度" |

## 4. Why 层：3 道选择题 + 1 道开放输入

### 4.1 设计理由

Why 层追问"为什么成立"，是最需要用户**自己组织因果链条**的层次。纯选择题的问题是：用户可能只是"认出"了正确答案，而不是"想通"了因果关系。

因此在 3 道选择题之后，增加 1 道开放输入题，让用户用自己的话解释一次原因。这既保留了选择题的低摩擦，又为原问回响的综合表达做了一次铺垫练习。

### 4.2 流程

```
3 道 4选1 单选题（选错重选直到正确）
        ↓
1 道开放输入题（AI 软判定，覆盖核心原因即通过）
        ↓
Why 层 completed
```

### 4.3 选择题界面（前 3 题）

与 How 层完全一致的布局，仅标题和色调不同。

```
┌────────────────────────────────────────────────┐
│  🔍 Why · 因果层 · 认知革命              1/3   │
├────────────────────────────────────────────────┤
│                                                │
│  🧙 老学者："尼安德特人个体比智人更强壮，       │
│            为什么最后是智人活下来了？"          │
│                                                │
│  ○ A. 因为智人运气好，尼安德特人遭遇了灾难      │
│                                                │
│  ○ B. 因为智人脑容量更大，更聪明                │
│                                                │
│  ○ C. 因为智人能组织更大规模的协作，            │
│       个体力量在群体协作面前不重要              │
│                                                │
│  ○ D. 因为智人更适应环境变化                    │
│                                                │
│                              [ 确认选择 → ]     │
└────────────────────────────────────────────────┘
```

### 4.4 开放输入题界面（第 4 题）

3 道选择题完成后，进入开放输入环节：

```
┌────────────────────────────────────────────────┐
│  🔍 Why · 因果层 · 认知革命              4/4   │
├────────────────────────────────────────────────┤
│                                                │
│  🧙 老学者："前面三题你都已经选对了。           │
│            现在，用你自己的话说说看——           │
│            为什么个体能力不是智人胜出的关键？"   │
│                                                │
│  💡 不需要长篇大论，1-3 句话说清               │
│     核心原因就好。                             │
│                                                │
│  ┌──────────────────────────────────────────┐  │
│  │                                          │  │
│  │  在这里输入你的解释...                    │  │
│  │                                          │  │
│  └──────────────────────────────────────────┘  │
│                                                │
│                    [ 提交回答 ]  [ 再想想 ]     │
└────────────────────────────────────────────────┘
```

### 4.5 开放输入题的 AI 软判定

| 判定维度 | 规则 |
|---------|------|
| 通过条件 | 回答触及 Why 层 Scope 标准的核心原因 |
| 反馈风格 | 鼓励性："你说到了 XX 这个关键点" |
| 未通过处理 | 提示"再想想"，但用户可选择"就这样提交" |
| 不强制重写 | 用户始终可以选择直接提交，不卡住 |

**软判定反馈示例（认知革命）：**

通过：
> 🧙 老学者："对，你抓住了'协作规模'这个核心——个体再强也敌不过大规模协作。这就是为什么尼安德特人虽然更强壮，却输给了智人。"

未通过（提示但不卡住）：
> 🧙 老学者："你的回答还差一点。想想看：智人赢在哪里？是个体更强，还是能组织更多的人？你可以再改改，也可以就这样提交。"

### 4.6 与 How 层的区别

| 维度 | How 层 | Why 层 |
|------|--------|--------|
| 题目焦点 | 机制与过程（"如何运作"） | 原因与因果（"为什么成立"） |
| 题量 | 3 道选择题 | **3 道选择题 + 1 道开放输入** |
| 对应 Scope 标准 | How 层标准 | Why 层标准 |
| 干扰项来源 | 机制类常见误解 | 因果类常见误解 |
| 界面色调 | 金绿（农场） | 大地色（废墟） |
| 输入环节 | 无 | **有，AI 软判定** |

### 4.7 Why 层出题示例（认知革命）

**对应 Scope 标准①：解释"为什么个体能力不是关键"**

**选择题（前 3 题）：**

| 题号 | 题干 | 正确选项核心 | 干扰项来源 |
|------|------|------------|-----------|
| 1 | 尼安德特人个体比智人更强壮，为什么最后是智人活下来了？ | 协作规模而非个体能力 | 误解"运气""脑容量""适应力" |
| 2 | 为什么说"人类更聪明"不能解释智人的胜出？ | 聪明是协作的结果，不是原因 | 误解"聪明是天生优势" |
| 3 | 如果虚构能力消失，智人社会最先崩溃的是什么？ | 大规模陌生人协作（信任基础） | 误解"工具使用""个体智力""资源分配" |

**开放输入题（第 4 题）：**

| 题干 | 判定核心 | 提示语 |
|------|---------|--------|
| "用你自己的话说说看——为什么个体能力不是智人胜出的关键？" | 回答是否触及"协作规模"或"大规模协作优于个体能力" | "想想看：智人赢在哪里？是个体更强，还是能组织更多的人？" |

## 5. System 层：卡片拖拽构建连接图

### 5.1 界面

```
┌──────────────────────────────────────────────────────┐
│  🌌 System · 系统层 · 认知革命                       │
├──────────────┬───────────────────────────────────────┤
│              │                                       │
│  卡片池       │      连接图画布                        │
│              │                                       │
│ ┌──────────┐ │   ┌────────┐    ┌────────┐           │
│ │🪙 货币    │ │   │🗣️ 认知  │───→│🤝 大规模│           │
│ │共同信任  │ │   │  革命   │    │ 协作   │           │
│ └──────────┘ │   └────────┘    └────────┘           │
│ ┌──────────┐ │        │                              │
│ │🏛️ 国家   │ │        ↓                              │
│ │想象秩序  │ │   ┌────────┐                          │
│ └──────────┘ │   │🪙 货币 │                          │
│ ┌──────────┐ │   │共同信任│                          │
│ │🏢 公司   │ │   └────────┘                          │
│ │法律虚构  │ │                                       │
│ └──────────┘ │  [ 拖拽卡片到画布，点击两张卡片连线 ]  │
│ ┌──────────┐ │                                       │
│ │📜 神话   │ │       已建立连接：2 / 3               │
│ │共同故事  │ │                                       │
│ └──────────┘ │              [ 完成连接 → ]            │
│ ┌──────────┐ │                                       │
│ │💼 上班   │ │                                       │
│ │现代场景  │ │                                       │
│ └──────────┘ │                                       │
│              │                                       │
└──────────────┴───────────────────────────────────────┘
```

### 5.2 交互规则

| 操作 | 说明 |
|------|------|
| 拖拽卡片 | 从左侧卡片池拖到右侧画布 |
| 连线 | 点击画布上的两张卡片，选择关系类型 |
| 关系类型 | 因果（→）/ 类比（≈）/ 包含（⊃）/ 演化（⇢） |
| 删除连接 | 点击连线删除 |
| 完成条件 | 建立 **≥3 条有效连接** |

### 5.3 卡片池设计

每个节点的 System 层提供 **5-6 张卡片**：

| 卡片类型 | 数量 | 内容 | 示例（认知革命） |
|---------|------|------|----------------|
| 本节点核心概念 | 2 | 本节点的核心知识点 | 认知革命、虚构故事 |
| 其他节点概念 | 2 | 可关联的其他节点知识 | 货币（共同信任）、国家（想象秩序） |
| 现实场景 | 1-2 | 可迁移的现实生活场景 | 上班（现代协作）、品牌（共同相信） |

### 5.4 连接有效性判定

系统自动判断用户建立的连接是否"有效"：

| 判定维度 | 说明 |
|---------|------|
| 逻辑成立 | 两张卡片之间确实存在因果关系、类比关系等 |
| 非重复 | 两条连接不能是同一对卡片的同一种关系 |
| 非琐碎 | "认知革命→认知革命"这类自连接无效 |

**判定方式**：预定义每对卡片的有效关系类型。用户建立的连接如果命中预定义关系，即为有效。不要求"唯一正确答案"——同一对卡片可能有多种有效关系。

### 5.5 完成反馈

建立 3 条有效连接后，老学者给出反馈：

> 🧙 老学者："你建立了 3 条连接。看看这张图——认知革命不是孤立的事件，它和货币、国家、甚至你每天上班都有关系。这就是 System 层的意义：把知识放回更大的系统里。"

### 5.6 数据结构

```typescript
interface SystemCard {
  id: string;
  label: string;          // 卡片标题
  description: string;    // 卡片描述
  category: 'core' | 'related' | 'realworld';
}

interface CardConnection {
  fromCardId: string;
  toCardId: string;
  relationType: 'cause' | 'analogy' | 'contain' | 'evolve';
}

interface SystemLayerData {
  cards: SystemCard[];                    // 5-6 张卡片
  validConnections: CardConnection[];     // 预定义有效连接
  requiredConnectionCount: number;        // 完成所需连接数（默认 3）
  scopeRef: string;                       // 对应 Scope 标准③
}
```

### 6.1 界面

```

├────────────────────────────────────────────────┤
│                                                │

│                  不同？"                        │
│                                                │
│  （这是你最初遇到的问题。现在，用你自己的话      │
│   回答吧——你已经从四个角度看过它了。）          │
│                                                │
│  ┌──────────────────────────────────────────┐  │
│  │                                          │  │
│  │  在这里输入你的回答...                    │  │
│  │                                          │  │
│  │                                          │  │
│  └──────────────────────────────────────────┘  │
│                                                │
│  💡 提示：想想你在 How/Why/System 层学到的     │
│     东西，把它们串起来。                       │
│                                                │
│                    [ 提交回答 ]  [ 再想想 ]     │
└────────────────────────────────────────────────┘
```

### 6.2 AI 软判定规则

| 判定维度 | 规则 |
|---------|------|
| 通过条件 | 回答覆盖 Scope 文档 3 条掌握标准中的 **≥2 条** |
| 反馈风格 | 鼓励性："你抓住了 XX 这个关键点" |
| 未通过处理 | 提示"再想想"，但用户可选择"就这样提交" |
| 不强制重写 | 用户始终可以选择直接提交，不卡住 |

### 6.3 软判定反馈示例

**覆盖 3 条标准（完全通过）：**

> 🧙 老学者："非常好。你提到了协作规模（标准①），解释了虚构如何促进协作（标准②），还用金钱举了例子（标准③）。你已经从四个角度理解了认知革命。"

**覆盖 2 条标准（通过）：**

> 🧙 老学者："不错。你抓住了协作规模和虚构促进协作这两个关键点。如果还能举一个现实例子就更完整了——不过你已经理解了核心。"

**覆盖 0-1 条标准（提示但不卡住）：**

> 🧙 老学者："你的回答还缺少一些关键点。想想看：智人和其他动物的差别到底在哪里？虚构故事在这个过程中起了什么作用？你可以再改改，也可以就这样提交。"

### 6.4 判定数据来源

AI 软判定直接使用 Scope 文档的 3 条掌握标准作为检查清单：

```typescript
interface FinalQuestionJudgment {
  criteria: {
    standard1Covered: boolean;   // 标准①是否覆盖
    standard2Covered: boolean;   // 标准②是否覆盖
    standard3Covered: boolean;   // 标准③是否覆盖
  };
  coveredCount: number;          // 覆盖数量（0-3）
  passed: boolean;               // coveredCount >= 2
  feedback: string;              // 鼓励性反馈
  canSubmitAnyway: boolean;      // 始终为 true，用户可强制提交
}
```

## 7. 对现有系统的影响

### 7.1 Scope 文档用途变化

Scope 文档的 3 条掌握标准**内容不变**，但用途从"LLM 判定 rubric"变成"出题依据"：

| 深度 | 旧用途 | 新用途 |
|------|--------|--------|
| How | LLM 按标准②判定自由回答 | **按标准②设计选项**，正确选项 = 标准②的表述 |
| Why | LLM 按标准①判定自由回答 | **按标准①设计选项**，正确选项 = 标准①的表述 |
| System | LLM 按标准③判定自由回答 | **按标准③设计卡片和连接目标** |
| 原问回响 | 综合判定 | **软判定**：覆盖 ≥2 条标准即通过 |

常见误解直接成为**干扰选项的来源**。

### 7.2 进度模型变化

| 深度 | 旧状态流转 | 新状态流转 |
|------|-----------|-----------|
| How | `locked → available → learning → completed` | `locked → available → completed`（无 learning） |
| Why | `locked → available → learning → completed` | `locked → available → completed`（无 learning，开放输入软判定不卡人） |
| System | `locked → available → learning → completed` | `locked → available → completed`（无 learning） |
| 原问回响 | `locked → available → completed` | 不变 |

How / Why / System 不再有 `learning` 状态。Why 层的开放输入虽然有一次 AI 判定，但因为是软判定（用户始终可以提交），不构成"学习中"状态——要么在做题，要么做完了。

### 7.3 PRD 需要更新的部分

| PRD 章节 | 变更内容 |
|---------|---------|
| §2.3 核心玩法模型 | How / Why 交互方式改为选项题，System 改为卡片拖拽 |
| §4.2.8 高阶问答引擎 | 整体重写为"分层交互引擎"：选项题 + 拖拽 + 软判定 |
| §4.2.10 认知进度模型 | How / Why / System 去掉 `learning` 状态 |
| §4.2.11 原问回响 | 改为 AI 软判定，覆盖 ≥2 条标准即通过 |
| §7 验收标准 | 更新为选项题、拖拽、软判定的验收项 |
