## 0. 初赛版本范围（参赛 Demo 优先级）

为了让初赛 Demo 在 30 秒内击中评委，本节明确锁死本版本要做、可选做、不做的范围。开发期间一切优先级冲突以本节为准。

### 0.1 初赛必须完成（P0，决定产品成立）

- 至少 1 个预制世界（《人类简史》）从入口村到第一个生物群系全部可玩
- 节点点击 → 学者化身像素小人移动 → NPC 对话框弹出 AI 提问
- AI 反馈卡：体现"已理解 / 还缺一点 / 提示 / 追问"四段式，让评委一眼看出这不是总结工具
- 迷雾消散动画：在 1080p 下流畅播放，作为 demo 视频前 10 秒的视觉钩子

### 0.2 初赛尽量完成（P1，加分项，做不完不影响演示）

- PDF 上传 → 自动生成世界（路演时作为"能力展示"短镜头，不作为主链路入口）
- 第二个预制世界（《经济学原理》第一章）
- 区域征服徽章

### 0.3 初赛不展示或弱化（P2，留到复赛）

- 世界列表 / 多世界切换
- 完整 6 项埋点（初赛只保留 `node_unlocked` 与 `session_complete`）
- 长期学习档案、积分、成就系统
- 申诉按钮、多模型协作

> 初赛阶段所有"上传任意 PDF 自动生成"的演示都从预制世界进入，避免现场生成等待与质量不可控带来的风险。

## 1. 需求背景

### 1.1 触发来源

AI 学习工具在 2024-2026 年迎来爆发，但绝大多数产品只解决了"信息获取"问题——总结、问答、生成课程——本质仍是把 AI 当作更聪明的"讲解员"。学习者拿到的依然是被动消费的内容，而非主动构建的理解。

### 1.2 用户当前现状与痛点

| 痛点 | 表现 | 后果 |
|---|---|---|
| 内容过载 | 收藏夹里堆满 PDF 与课程，真正读完的不到 10% | 知识焦虑，越学越虚 |
| 被动消费 | 读完一本书或听完一门课，能复述但不会用 | 理解停留在 What 层 |
| 缺乏反馈 | 不知道自己哪里没懂，也没人追问 | 自我评估失真 |
| 学习无趣 | 严肃学习场景缺乏正反馈机制 | 难以坚持 |

证据强度：`[PM 假设 + 行业观察]`，正式版 PRD 上线前需通过 30 人定性访谈验证。

### 1.3 不解决的后果

如果 AI 学习产品继续停留在"总结/讲解"形态，用户会快速对 AI 学习工具产生疲劳，市场窗口被 ChatGPT/Claude 等通用对话产品吃掉。差异化窗口正在快速关闭。

### 1.4 为什么现在可行

大模型已经具备稳定的语义理解、结构抽取、问题生成和反馈生成能力，学习资料可以从"静态内容"转化为"动态认知交互系统"。过去的知识图谱只能组织信息，现在的 AI 可以判断用户的理解层级、定位误区，并持续生成下一步问题，这让探索式学习从界面创意变成可落地的产品机制。

## 2. 产品目标

### 2.1 核心目标

> 让学习像探索未知世界一样发生。

用户把任意一份学习资料转化为可导航的知识世界，AI 先用问题诊断理解，再给出分层反馈与必要讲解，帮助用户跨过卡点。

### 2.1.1 产品定位

Knowledge World 的定位不是 AI 学习工具，而是认知探索系统：

```text
Knowledge → World → Exploration → Cognition Training
```

它关注的不是"把知识讲得更快"，而是帮助用户看见自己的理解边界，并通过一次次探索把边界向外推进。

### 2.2 量化目标（参赛 MVP 阶段）

| 维度 | 目标 |
|---|---|
| Demo 演示效果 | 上传 PDF 后 ≤90 秒生成可探索世界地图 |
| 核心交互完成率 | 测试用户首次会话内能完成 ≥3 个节点的解锁 |
| 视觉记忆点 | 路演评委观看 30 秒后能复述"知识迷雾"机制 |
| 反馈可信度 | AI 追问 - 用户回答 - 诊断反馈 - 解锁判断的链路可在 demo 中跑通 3 轮以上 |

### 2.3 非目标（明确不做）

- 不做登录系统（MVP 阶段匿名使用，用户数据存 localStorage）
- 不做长期用户成长体系（积分、段位、皮肤等）
- 不做社交分享（评委 demo 阶段无关）
- 不做多人协作探索
- 不做移动端适配（仅 Web 桌面端）

## 3. 用户与场景

### 3.1 目标用户画像

| 画像 | 描述 | 决策权重 |
|---|---|---|
| **求知型自学者** | 有学习目标但缺方法的成年人，例如刚入门读《人类简史》的读者 | 主要 |
| **应试备考者** | 需要构建知识体系的考研/考证用户 | 次要 |
| **教育评委 / 比赛评审** | 关心创新性与产品完整度 | MVP 阶段优先服务 |

不服务：寻找快速答案的用户、需要权威讲解的用户、儿童/低龄学习者。

### 3.2 核心场景排序

| 优先级 | 场景 | 描述 |
|---|---|---|
| P0 | 上传一本书生成世界 | 用户上传《人类简史》PDF，90 秒内得到一张可探索的像素风世界地图 |
| P0 | 节点探索与追问 | 用户点击节点，学者化身走过去，触发 AI 苏格拉底式提问 |
| P0 | 答题反馈与解锁 | 用户输入回答，AI 诊断理解缺口，给出分层反馈或最小讲解，再根据状态点亮节点 |
| P1 | 区域征服反馈 | 完成一个层级（如认知大草原）所有节点后，弹出区域成就徽章 |
| P2 | 切换世界 | 同一用户已生成多个世界，可在世界列表间切换 |

### 3.3 使用前提

- 用户提供合法的学习资料文件（PDF / TXT / Markdown，单文件 ≤20MB）
- 用户具备基本中文阅读与输入能力
- 浏览器支持 WebGL 与 Canvas

## 4. 功能需求

### 4.1 功能总览表

| # | 模块 | 功能描述 |
|---|---|---|
| 1 | 资料上传 | 用户在首页上传 PDF/TXT/MD 文件。系统接收文件后展示上传进度、解析进度、世界生成进度三段式提示。失败时提供原因（文件过大/格式不支持/解析失败）和重试入口。生成成功后自动跳转到世界地图页。 |
| 2 | 世界生成引擎 | 后端解析文件文本，分块后调用 LLM 抽取核心概念，按四层认知模型（What/How/Why/System）聚类并生成不超过 20 个节点的世界结构。每个节点附带：节点名、所属层级、父节点、相邻节点、对应原文片段、像素图标类型。 |
| 3 | 像素风世界地图 | 渲染一张 16:9 的像素风地图。地图分四个生物群系（对应四层）。学者像素化身初始位于"入口村"。未解锁节点上方显示问号标记，已解锁节点显示其图标。第④层（System 层）默认覆盖紫蓝色雾气贴图。 |
| 4 | 节点探索交互 | 用户点击地图上的相邻节点（与已解锁节点通过路径连接的节点）。学者化身按路径走过去（约 1.5 秒像素小人移动动画）。到达后弹出 NPC 对话框，触发 AI 提问。 |
| 5 | 苏格拉底提问引擎 | 针对当前节点，AI 生成 1 个核心提问 + 准备 2 个追问。提问不直接给结论、不提供选项、不暴露原文答案。问题风格固定为"为什么 / 如果...会怎样 / 这与...有什么关系"。 |
| 6 | 用户回答与诊断反馈 | 用户在对话框输入文字回答（≤500 字）。AI 先判断回答覆盖了哪些理解层级（概念/逻辑/迁移），再输出反馈卡：已理解内容、缺失点、必要提示或最小讲解、下一问。最多 3 轮交互后强制判定。 |
| 7 | 迷雾消散与解锁 | 节点达到"已掌握"或"迁移应用"后，触发 2-3 秒迷雾消散动画。已掌握：当前节点绿色光晕增强 → 周围迷雾消散 → 邻接节点变彩。迁移应用：节点出现紫色光环，代表用户能把知识迁移到新情境。 |
| 8 | Knowledge Progress Model | 每个节点具有五种学习状态：未探索、已访问、理解中、已掌握、迁移应用。地图右上角显示"已掌握 X / 共 Y 节点"，区域完成度以状态颜色呈现，不显示具体百分比，避免分数焦虑。 |
| 9 | 区域征服徽章 | 当一个生物群系内所有节点全部解锁，触发全屏祝贺弹窗：徽章图标 + "你已征服认知大草原" + 关闭按钮。徽章保存在用户档案中。 |
| 10 | 世界列表 | 顶部导航提供"我的世界"入口，列出用户已生成的所有世界（缩略图 + 名称 + 进度）。可切换、可删除。 |

### 4.2 模块详情

#### 4.2.1 资料上传（# 1）

**原型示意**

```
+------------------------------------------------------+
|  Knowledge World                            [我的世界]|
+------------------------------------------------------+
|                                                       |
|        把任何资料，变成一个可探索的世界                |
|                                                       |
|       +---------------------------------------+       |
|       |       拖拽 PDF/TXT/MD 到这里           |       |
|       |             或点击选择                 |       |
|       +---------------------------------------+       |
|                                                       |
|       支持：PDF / TXT / Markdown，≤ 20 MB             |
|                                                       |
|       [试试预制世界：人类简史 / 经济学原理]            |
|                                                       |
+------------------------------------------------------+
```

- **业务逻辑**：文件上传 → 后端校验大小与格式 → 文本抽取 → 异步生成世界 → 推送进度事件 → 完成后跳转。
- **交互逻辑**：上传过程显示三段式进度（上传 / 解析 / 生成），每段独立百分比；任一段失败弹出错误卡片，提供"重试"和"换一个文件"。
- **规则约束**：
  - 文件大小：≤ 20 MB
  - 格式白名单：`.pdf` / `.txt` / `.md`
  - 文件名长度：≤ 100 字符
  - 同一用户并发生成任务：≤ 1（防止 LLM 调用爆炸）
- **预制世界**：MVP 至少预置 2 个生成好的世界（《人类简史》《经济学原理（曼昆）第一章》），首屏即可点击试玩，绕开等待期流失。

#### 4.2.2 世界生成引擎（# 2）

**输入**：解析后的纯文本（按章节切块）。
**输出**：JSON 格式的世界结构。

**世界结构 JSON 示例**：

```json
{
  "world_id": "w_2026_xxx",
  "title": "人类简史",
  "biome_theme": "history-civilization",
  "layers": [
    {
      "layer": "what",
      "biome_name": "认知大草原",
      "nodes": [
        {
          "id": "n_001",
          "name": "认知革命",
          "icon_type": "cave_painting",
          "neighbors": ["n_002", "n_003"],
          "source_excerpt": "约七万年前，智人开始..."
        }
      ]
    }
  ],
  "start_node_id": "n_001"
}
```

- **业务逻辑**：
  1. 文本切块（每块 ≤ 1500 token，重叠 200 token）
  2. LLM 抽取候选概念（每块输出 3-5 个）
  3. 概念去重 + 聚类（向量相似度 + LLM 二次合并）
  4. 按四层认知模型分类（提示词引导 LLM 判断每个概念属于 What/How/Why/System）
  5. 节点关系生成（依据原文邻近度与 LLM 语义判断）
  6. 节点总数硬上限 20，超出时按重要性裁剪
- **规则约束**：
  - 节点总数：≤ 20
  - 每层节点数：3-7（不平衡时按 What 多 / System 少自动裁剪）
  - 节点名长度：≤ 12 字符（保证地图可读性）
- **异常处理**：
  - LLM 调用失败 3 次 → 任务标记失败，用户可重试
  - 文本过短（<500 字）→ 拒绝生成，提示"资料过短"

#### 4.2.3 像素风世界地图（# 3）

**原型示意**：直接复用 `assets/world_map_sapiens.jpg` 作为视觉基线（路演中作为"上传《人类简史》后的生成结果"展示）。

- **业务逻辑**：前端拿到世界 JSON 后，按层级渲染四个生物群系到固定 16:9 画布上。每层节点按力导向布局算法在该层区域内自动排布，避免重叠。
- **交互逻辑**：
  - 鼠标悬停节点 → 显示节点名 tooltip
  - 点击已解锁节点 → 居中视图，但不触发提问
  - 点击邻接（可解锁）节点 → 学者化身移动 + 触发提问
  - 点击远处未点亮节点 → 提示"先解锁路径上的节点"
- **规则约束**：
  - 地图分辨率：1366 × 768 基线，≥ 1920px 自动放大
  - 视图：固定俯视 3/4 视角，不支持自由缩放（MVP 阶段）
  - 像素风格基线：暖黄（What 层）→ 金绿（How 层）→ 石灰（Why 层）→ 紫蓝（System 层）
- **状态**：
  - 未探索：灰色迷雾 + 问号，表示用户尚未进入该节点
  - 已访问：节点轮廓可见，表示用户到达过，但尚未完成理解
  - 理解中：黄色弱光，表示用户有部分理解，需要提示或追问
  - 已掌握：绿色亮光，表示用户已理解核心概念与逻辑
  - 迁移应用：紫色光环，表示用户能将知识迁移到新情境
  - 当前所在节点：脚下出现像素光圈

#### 4.2.4 苏格拉底提问引擎（# 5）与诊断式反馈（# 6）

> 这是产品的认知核心，也是最容易翻车的地方，必须最严格设计。

核心机制定义为 **AI Cognitive Engine**：

```text
AI 不是聊天，而是在做三件事：
1. 判断用户处于哪个认知层级
2. 找出当前最大理解误区
3. 生成下一步最优问题
```

这个机制让产品区别于普通 ChatGPT 对话。ChatGPT 主要响应用户问题，而 Knowledge World 主动维护一条 Socratic Path：系统知道当前节点要训练什么、用户卡在哪里，以及下一问应该把用户推向哪个理解层级。

核心交互原则为：

> 先提问，再反馈；先诊断，再讲解；先让用户思考，再给答案。

AI 不应在用户进入节点后立刻灌输结论，但当用户明显不知道、回答严重偏离或连续两轮卡住时，需要给出最小必要讲解。讲解必须服务于下一次思考，而不是替用户完成理解。

- **提问生成 Prompt 骨架**：
  ```
  你是一名苏格拉底式导师。学习者刚走到节点【{node_name}】。
  原文片段：{source_excerpt}
  这个节点属于【{layer}】层级（What/How/Why/System）。
  请生成 1 个核心提问 + 2 个追问。
  规则：
  - 第一问不直接给结论
  - 不暴露原文答案
  - 不提供选项
  - 用"为什么 / 如果...会怎样 / 这与...有什么关系"句式
  ```
- **诊断反馈 Prompt 骨架**：
  ```
  原文要点：{source_key_points}
  学习者回答：{user_answer}
  当前轮次：{round}
  请输出 JSON：
  {
    "cognitive_level": "what" | "how" | "why" | "system",
    "covered_dimensions": ["concept" | "logic" | "transfer"],
    "main_misconception": "当前最大的理解误区",
    "missing_points": ["缺失要点 1", "缺失要点 2"],
    "next_best_question": "下一步最优问题",
    "feedback_level": "reinforce" | "hint" | "minimal_explain",
    "feedback_card": {
      "understood": ["用户已经理解的点"],
      "missing": ["还缺的关键点"],
      "guidance": "根据 feedback_level 输出强化反馈、提示或最小必要讲解",
      "next_question": "使用 next_best_question 继续推进理解"
    },
    "node_state": "unexplored" | "visited" | "learning" | "mastered" | "transfer"
  }
  ```
- **业务逻辑**：每个节点最多 3 轮交互。
  - 用户基本答对：输出强化反馈，节点进入 `mastered`，触发完整迷雾消散
  - 用户能举一反三：节点进入 `transfer`，出现紫色光环，区域成就权重更高
  - 用户答得模糊：输出提示与追问，节点进入 `learning`
  - 用户完全不知道：输出最小必要讲解，再给一个更低门槛追问
  - 连续 3 轮未达到 `mastered`：节点保持 `learning`，允许继续探索邻接节点，但该节点不计入区域徽章
- **规则约束**：
  - 用户回答长度：≤ 500 字
  - 单次诊断超时：5 秒，超时回退为"理解中 + 留追问"
  - LLM 输出非合法 JSON：重试 1 次，仍失败则按 `learning` 处理（保护用户体验）
  - 最小必要讲解长度：≤ 120 字，必须包含一个后续追问

**反馈卡样式示例**：

```
理解状态：接近掌握

你已经理解：
- 农业革命改变了人类获取食物的方式
- 定居生活让人口增长成为可能

还缺一点：
- 农业革命也带来了劳动强度上升、社会等级和财产观念

提示：
当粮食可以储存后，谁来分配粮食、谁拥有土地、谁承担劳动，都会变成新的社会问题。

追问：
如果农业革命让粮食更多，为什么作者反而说它未必让个体更幸福？
```

#### 4.2.5 Knowledge Progress Model（# 8）

学习状态系统让 MVP 从"演示效果"变成"可持续学习系统"。每个节点不再只有解锁/未解锁两种状态，而是记录用户理解如何变化。

| 状态 | 地图表现 | 进入条件 | 后续影响 |
|---|---|---|---|
| 未探索 | 灰色迷雾 + 问号 | 用户尚未进入节点 | 只能从相邻已访问节点进入 |
| 已访问 | 节点轮廓可见 | 用户点击并进入过节点 | 可再次挑战，不计入掌握 |
| 理解中 | 黄色弱光 | 用户答到部分要点，但仍有明显缺口 | 可继续探索邻接节点，但区域徽章不计入 |
| 已掌握 | 绿色亮光 | 用户覆盖核心概念与主要逻辑 | 计入区域完成度，解锁邻接节点 |
| 迁移应用 | 紫色光环 | 用户能举例、类比或迁移到新场景 | 计入高阶掌握，影响最终成就文案 |

状态流转如下：

```text
未探索 → 已访问 → 理解中 → 已掌握 → 迁移应用
                 ↘ 最小讲解 + 追问 ↗
```

产品侧不向用户展示"72% 掌握度"这类分数，而是通过颜色、迷雾、光环和路径变化让用户感知自己的学习推进。

#### 4.2.6 迷雾消散动画（# 7）

> 这是路演 Demo 视觉冲击力的最关键节点，必须做好。

- **动画分镜**：
  1. T0：节点中心闪光（白光，0.3 秒）
  2. T0+0.3s：光圈以节点为中心向外辐射（1 秒，半径覆盖到邻接节点）
  3. T0+0.5s：紫蓝色迷雾粒子向外扩散并淡出（1 秒）
  4. T0+1.5s：邻接节点从灰转彩，浮现问号标记（弹性缩放动画 0.4 秒）
  5. T0+2s：右上角进度条 +1，伴随轻量音效（可选）
- **技术实现要点**：使用 Canvas 或 PixiJS，迷雾以粒子贴图驱动，邻接节点用 CSS 关键帧 / GSAP。

## 6. 风险与降级方案

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| LLM 评估不一致（用户答对被判错） | 高 | 高 | 引入"通过 + 留追问"软判定，避免硬性拒绝；提供"我觉得我答对了"申诉按钮（MVP 后接入） |
| AI 反馈过度，重新变成讲课工具 | 中 | 高 | 限制最小讲解长度 ≤120 字；每次讲解后必须接追问；反馈卡固定包含"你已理解 / 还缺一点 / 追问" |
| 生成耗时过长（>2 分钟） | 中 | 高 | 预制世界兜底；生成期展示"探险家在绘制地图"动画掩盖等待 |
| 像素地图布局重叠 | 中 | 中 | 节点数硬上限 20；力导向算法 + 重叠检测重排 |
| LLM 配额不足或超时 | 中 | 中 | MVP 用 OpenAI API，复赛前评估切换火山方舟/DeepSeek；后端封装 LLMAdapter 接口，切换模型只改配置；预备双通道降级 |
| 评委 30 秒内 get 不到产品价值 | 中 | 极高 | Demo 视频前 10 秒必须出现"迷雾消散"镜头 |

## 8. 验收标准

MVP 必须同时满足以下条件方可视为可参赛：

- 任意上传一本 50 页以内 PDF，能在 90 秒内生成可探索世界

- 单个会话内可完成至少 5 个节点的"提问→回答→诊断反馈→解锁"完整链路
- 用户完全不知道时，系统能输出 ≤120 字的最小必要讲解，并继续给出一个追问
- 迷雾消散动画在 1080p 下流畅播放（≥30 帧）
- 评委盲测：3 名非技术人员看完 demo 视频后能独立复述"知识迷雾"和"学者化身探索"两个核心机制

## 10. LLM 选型策略（补充）

MVP 阶段 LLM 选型采用"先跑通再优化"策略：

| 阶段 | 模型 | 用途 | 理由 |
|---|---|---|---|
| MVP 验证（06-16 ~ 07-06） | OpenAI GPT-4o | 世界生成 + 提问 + 诊断反馈 | 效果最稳定，Prompt 调试效率最高 |
| 初赛提交前（07-06 ~ 07-14） | 评估后决定 | 可能切换到火山方舟（豆包）或 DeepSeek | 降低成本 + 国内访问稳定性 |
| 复赛阶段（07-20 ~ 08-10） | 待定 | 根据初赛反馈 + 评委建议调整 | 可能需要多模型协作（生成用大模型，评估用小模型） |

架构要求：后端封装统一的 `LLMAdapter` 接口，所有 LLM 调用通过适配器，切换模型只需改配置，不动业务代码。
